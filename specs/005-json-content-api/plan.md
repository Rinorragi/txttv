# Implementation Plan: JSON Content API & Two-API Architecture

**Branch**: `005-json-content-api` | **Date**: February 24, 2026 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/005-json-content-api/spec.md`

## Summary

Replace the current monolithic APIM architecture (11 × 679-line full-page fragments with duplicated CSS/JS/content) with a two-API pattern: a **Page API** serving a single shared HTML template and a **Content API** serving structured JSON from minimal content fragments. Content files migrate from plain `.txt` to structured `.json`, enabling JavaScript-based rendering via htmx/fetch. Content APIM fragments mirror source JSON files as closely as possible for local development parity. A PowerShell-based local HTTP server enables the same fetch workflow during development.

## Technical Context

**Primary Implementation**: APIM policy fragments (content fragments + page template fragment + routing policies)
**Language/Version**: PowerShell 7+ (conversion scripts, dev server), JavaScript ES6 (client-side renderer)
**Infrastructure as Code**: Bicep (Azure-native IaC)
**Cloud Platform**: Azure (required for APIM/AppGW/WAF integration)
**APIM Policies**: Content routing policy (JSON dispatch), updated page routing policy (shared template), content fragments (minimal JSON wrapper), page template fragment (HTML shell with htmx)
**Primary Dependencies**: htmx 2.0.8 (CDN: jsdelivr), APIM policy XML schema, PowerShell System.Net.HttpListener
**Storage**: Content stored as JSON files in repository (`content/pages/`), deployed as APIM policy fragments (no external storage service)
**Testing**: APIM policy XML validation, JSON schema validation (PowerShell), Pester tests for conversion scripts, integration tests for page navigation, WAF rule tests
**Target Platform**: Azure APIM (primary), local PowerShell HTTP server (development)
**Project Type**: apim-policy-centric
**Performance Goals**: Content fragments < 2 KB each (down from 16-17 KB monolithic), conversion of all 11 pages < 10 seconds, single fetch per page view
**Security Requirements**: WAF rules for SQL injection/XSS on Content API, APIM rate limiting on both APIs, JSON content sanitization in client renderer, input validation on pageNumber parameter
**Scale/Scope**: 11 pages (100-110), 1 page template fragment, 11 content fragments, 1 content routing policy, 1 updated page routing policy, 1 conversion script, 1 dev server script, 1 client-side renderer

- [x] Components independently deployable — content fragments deployable independently from page template

- [x] Contract tests for policy inputs/outputs defined — JSON schema validation for content, HTML structure for page template
- [x] Integration tests for WAF rules planned — extend existing WAF tests for Content API endpoint

### Principle V: Deployment Automation
- [x] CI/CD pipeline defined as code — existing pipeline extended for new artifacts
- [x] Automated validation gates specified — JSON schema validation + XML well-formedness + fragment size limits
- [x] Rollback procedure documented — content fragments version-controlled, rollback = redeploy previous commit

### Complexity Justification Required?
- [x] No violations — design aligns with all five principles. The local HTTP server is a **complexity addition** but justified: constitution v1.2.2 says "no server required for static content" but the two-API architecture requires fetch requests which fail on `file://` protocol. PowerShell `System.Net.HttpListener` is the minimal solution without Node.js dependencies.
